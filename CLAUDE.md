# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Market data proof-of-concept FPGA design for the **AMD/Xilinx KR260 Starter Kit** (Zynq UltraScale+ MPSoC, part `xck26-sfvc784-2LV-c`). The design receives 10G Ethernet frames via an SFP port (Xentech Robotics Card), processes them in PL logic, and transfers packets to the Zynq PS via AXI DMA.

## Build

This is a Vivado project reconstructed from a TCL script. There is no Makefile — the entire project is recreated by sourcing the script in Vivado.

**Recreate the project from Vivado Tcl Shell:**
```tcl
source mktdata_poc.tcl
```

**From the command line:**
```bash
vivado -mode tcl -source mktdata_poc.tcl
```

The script creates a new project directory `./mktdata_poc/`. After creation, launch synthesis and implementation manually in the Vivado GUI or via Tcl (`launch_runs synth_1`, `launch_runs impl_1`).

**Generate bitstream:**
```tcl
launch_runs impl_1 -to_step write_bitstream -jobs 4
```

## Source Files

All RTL lives in `ip/`. The project has no simulation testbenches.

| File | Description |
|------|-------------|
| `ip/xgmii_includes.vh` | XGMII control character constants (S/T/E/I), preamble patterns, and synthesizable inline CRC-32 functions (crc1B–crc8B) |
| `ip/xgmii2axis.v` | XGMII → AXI-Stream adapter; strips preamble/FCS, checks CRC, outputs two parallel streams: `zy_*` (→ Zynq PS via DMA) and `lv_*` (→ NI FPGA IP) |
| `ip/axis2xgmii.v` | AXI-Stream → XGMII adapter; adds preamble/FCS, implements Deficit Idle Count (DIC) for 802.3 IPG compliance |
| `ip/my_state.v` | Simple 64-bit accumulator with `control[1:0]`-driven add/clear; exposed as a custom AXI-lite register block |
| `ip/kr260_starter_kit_wrapper.v` | Auto-generated Vivado block design wrapper (top-level module); do not edit manually |
| `ip/NiFpgaAG_poc_ip.v` | NI LabVIEW FPGA auto-generated IP — the market data parsing logic |
| `ip/NiFpgaIPWrapper_poc_ip.vhd` | VHDL wrapper for the NI IP; exposes AXI-Stream ports for CMD/DEBUG/MDEBUG streams |

## Architecture

```
SFP (10G) ─── xxv_ethernet IP ─── XGMII (156.25 MHz, 64-bit)
                                       │
                                  xgmii2axis.v
                                  ┌────┴──────┐
                                  zy_* stream  lv_* stream
                                  (→ AXI DMA   (→ NI FPGA IP
                                   → Zynq PS)   → CMD/DEBUG/MDEBUG
                                                  AXI-S outputs)
```

The block design (`kr260_starter_kit`) includes:
- **Zynq UltraScale+ PS** — ARM cores, DDR
- **xxv_ethernet** — 10G/25G Ethernet subsystem providing XGMII interface
- **AXI DMA** — transfers Ethernet frames between PL and PS memory
- **AXI GPIO / IIC** — SFP management and LEDs
- **Proc Sys Reset** — reset sequencing
- **TX/RX data FIFOs** — AXI-Stream clock-crossing FIFOs between DMA and XGMII adapters

## Clock Domains

| Domain | Frequency | Used for |
|--------|-----------|----------|
| `Clk40MhzDerived5x2B00MHz` | 100 MHz | AXI/PS control, AXI DMA |
| `Clk40MhzDerived168x43B56_28MHz` | 156.25 MHz | XGMII, xgmii2axis, axis2xgmii, NI FPGA IP data path |

## Constraints

- `default.xdc` — bitstream compression, fan enable pin (A12)
- `Xentech.xdc` — all GTH/SFP pin assignments for the Robotics Card; false paths on IIC/LED outputs. Use this file when targeting the Xentech SFP hardware configuration.

## Key RTL Notes

- **CRC-32**: Computed inline in `xgmii_includes.vh` using combinational XOR expressions per byte count variant (crc1B–crc8B). These are parallel CRC functions, not LFSR-style.
- **TUSER**: In both adapters, `TUSER[0] = 1` signals a **good** frame (CRC pass + valid terminator). This is inverted from typical error conventions.
- **Lane alignment**: XGMII can start frames on lane 0 (byte 0) or lane 4 (byte 4). Both `axis2xgmii` and `xgmii2axis` handle both start positions.
- **DIC**: `axis2xgmii` maintains a 2-bit Deficit Idle Count to compensate for shortened IPGs when frames terminate mid-quadword.
- **NI FPGA IP**: The `.v` file is generated by LabVIEW FPGA IP Export and should not be hand-edited. Re-export from LabVIEW if the processing logic changes.
